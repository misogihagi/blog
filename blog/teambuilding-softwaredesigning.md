# チームビルディングとソフトウェア設計の共通点

組織もプロダクトも、スケールするためには「内部の最適化」がまず必要になる。

例えば、仕事を任せるにしてもいきなり丸投げにすると高確率で失敗するし、何を任せるべきか、どうすれば達成できるかを考えて発注すれば、例え失敗に近いところで行ってもなぜ失敗したか腑に落ちる。

内部のメンバーだけで運用していると、暗黙の了解や人に合わせた最適解によって組織やシステムはどんどん密結合になり、属人化される。
ここに外部の人が入ることで、以下のサイクルが回り始める。

インターフェイスの明確化： 「誰が・何を・どう渡せば仕事が完了するのか」という境界線（インターフェイス）が言語化される。
拡張性の獲得： 結合部が整理されるため、別のチームや新しい人をプラグインのように簡単に追加できるようになり、爆発的なスケールが可能になる。

一方で、このインターフェイスの整備を怠ったり、外部の新しい血を入れることを拒絶したりすると、組織やプロダクトは大きくなることができなくなり、全体像を知っている古参のメンバーしか手を出せなくなり、ボトルネックになる。
結果として、スケールしないどころか、競争に負ける。

つまり成長するためには、以下のサイクルに
1. 内部が頑張る（内部の最適化・ドキュメント化・標準化）
2. 内部の人間が自らを客観視し、属人化している業務やコードを徹底的に整理する。
3. 外部を取り入れる（オープン化・外部人材の登用）
4. あえて外部の視点や人材を積極的に組み込むことで、強制的にインターフェイスを整理させる。

プロダクト開発における「モノリシック（巨大で一体型）な構造」を、いかに「マイクロサービス（独立した機能が連携する構造）」に分割してスケールさせるかという思想と、組織マネジメントにおけるチームビルディングの思想は本質的に同じだと言える。
